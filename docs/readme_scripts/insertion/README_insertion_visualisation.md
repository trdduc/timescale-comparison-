# Benchmark Visualisation Tool

A plotting script that reads the metrics CSV files produced by the parallel loader and generates publication-quality charts comparing PostgreSQL vs TimescaleDB insertion performance.

## What it does

The script reads one or two benchmark CSV files from a `metrics/` directory, combines them, and produces three PNG charts saved to a `docs/` directory:

| Chart file | What it shows |
|---|---|
| `insertion_stability.png` | Insertion rate (rows/sec) as the table grows — reveals performance degradation under index pressure |
| `throughput_over_time.png` | Data throughput (MB/s) over the duration of the test — reveals IO consistency |
| `latency_comparison.png` | Average batch commit latency as the table grows — reveals how each DB handles increasing volume |

All three charts overlay both databases on the same axes using distinct colors, making direct comparison straightforward.

## Directory structure expected

```
project/
├── insertion_visualisation.py
├── metrics/
│   ├── timescale_bench.csv      # metrics from TimescaleDB run
│   └── postgres_bench.csv       # metrics from PostgreSQL run
└── docs/                        # created automatically if missing
```

The `metrics/` directory and CSV file names are hardcoded. The script warns and skips gracefully if one of the files is missing — you can run it with only one database's data and it will still produce valid charts.

> **Where do the CSV files come from?**
> They are generated by the parallel loader using its `--out-metrics` flag. See the loader's README for details.

## Usage

```bash
python insertion_visualisation.py
```

No arguments. All configuration is done by editing the constants at the top of the script.

On success, output looks like:

```

[+] Saved: docs/insertion_stability.png
[+] Saved: docs/throughput_over_time.png
```

## Configuration (edit in script)

All tuneable values are defined at the top of the file:

| Variable | Default | Description |
|---|---|---|
| `METRICS_DIR` | `'metrics'` | Directory where input CSV files are read from |
| `DOCS_DIR` | `'docs'` | Directory where PNG charts are saved |
| `DB_LABELS` | `{'timescale_bench.csv': 'TimescaleDB', 'postgres_bench.csv': 'PostgreSQL'}` | Maps each CSV filename to the label shown in chart legends |

To add a third database, or rename the input files, just extend the `DB_LABELS` dictionary:

```python
DB_LABELS = {
    'timescale_bench.csv': 'TimescaleDB',
    'postgres_bench.csv': 'PostgreSQL',
    'cockroach_bench.csv': 'CockroachDB'   # add more as needed
}
```

## Input CSV format

Each file in `metrics/` must contain at least these columns (produced automatically by the parallel loader):

| Column | Used in |
|---|---|
| `total_rows` | Plots 1 and 3 (x-axis) |
| `period_rate_rows_sec` | Plot 1 (y-axis) |
| `elapsed_sec` | Plot 2 (x-axis) |
| `period_throughput_mb_sec` | Plot 2 (y-axis) |
| `avg_batch_latency_s` | Plot 3 (y-axis) |

Rows with missing or non-numeric values in `total_rows` or `period_rate_rows_sec` are dropped automatically before plotting.

## Output chart details

### `insertion_stability.png` — Insertion Rate vs Volume
- X-axis: cumulative rows inserted, formatted in millions (e.g. `1.0M`, `2.5M`)
- Y-axis: instantaneous rows/sec during each reporting period
- Markers shown at ~10% intervals to avoid clutter on dense lines
- Useful for spotting the point where index growth starts degrading insert speed

### `throughput_over_time.png` — Throughput vs Time
- X-axis: elapsed seconds since the job started
- Y-axis: MB/sec during each reporting period
- No markers (cleaner for time-series comparison)
- Useful for identifying IO bottlenecks or throttling

### `latency_comparison.png` — Batch Latency vs Volume
- X-axis: cumulative rows, formatted in millions
- Y-axis: average `COPY` commit latency in seconds per batch
- Useful for understanding how commit cost scales with table size

All charts are saved at **300 DPI**, suitable for inclusion in papers or reports.
